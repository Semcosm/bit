# bit

> A new systems programming language built around LLVM IR.
>
> **bit** 的目标是成为一个以 **LLVM IR** 为核心后端依赖、面向未来系统软件开发的基础编程语言。  
> 它希望在全平台上继承 LLVM 的优化能力、目标平台支持与工具链生态，并在语言层面重新思考 **性能、安全性、可维护性、可移植性** 之间的平衡。

---

**当前阶段：早期构想 / 原型设计阶段**

bit 目前仍处于项目早期，核心目标、语言边界、类型系统、语义设计、编译流程与自举路线正在持续收敛中。

## bit 是什么

bit 是一个面向系统编程的语言项目。  
它不是 LLVM 的替代品，而是建立在 LLVM 之上的语言与工具链。

bit 的核心思路是：

1. **语言前端负责表达能力、类型系统、语义约束与开发体验**
2. **LLVM IR 负责统一中间表示、优化管线与多平台代码生成**
3. **语言设计从一开始就为性能、可控性与长期演进服务**

换句话说，bit 想做的不是“再造一个语法糖语言”，而是尝试构建一个真正适合长期演进的基础语言。

---

## 项目目标

bit 的长期目标包括但不限于：

- 提供接近底层、但比传统 C 风格更现代的系统编程体验
- 尽可能直接地利用 LLVM 的优化与后端能力
- 在 Linux / Windows / macOS 及更多架构上保持一致的编译模型
- 建立清晰、严格、可演进的语言规范
- 为未来自举、自托管与完整工具链生态打下基础

---

## 设计原则

### 1. 性能优先，但不牺牲语言结构

bit 必须能够生成高质量机器码，并保持对性能行为的强控制力。  
抽象不能默认意味着高代价，零成本抽象应当是重要目标。

### 2. 明确优于隐式

语言语义、类型行为、内存模型、错误处理策略都应尽量清晰。  
bit 倾向于减少“看起来方便、实际上难以推断”的隐式机制。

### 3. 以 LLVM IR 为核心工程依托

bit 不重复发明成熟后端，而是把精力集中在语言本身与前端架构上。  
LLVM 提供的平台支持、优化能力与工具链兼容性，是 bit 的关键基础设施。

### 4. 可自举、可收敛、可维护

bit 不追求一开始就“什么都有”。  
项目会优先建立最小闭环，再逐步扩展语法、类型系统、标准库与工具链。

### 5. 系统语言，而不是脚本语言

bit 的目标不是替代 shell、Python 或胶水语言生态。  
它更关注：

- 编译型工作流
- 明确的类型系统
- 可控的资源管理
- 长期可维护的大型工程

---

## 为什么选择 LLVM IR

选择 LLVM IR 作为核心后端依赖，主要出于以下考虑：

- **成熟的优化能力**：LLVM 已经拥有非常强的优化管线
- **广泛的平台支持**：能覆盖主流操作系统与 CPU 架构
- **丰富的工具链生态**：便于接入汇编、链接、调试、反汇编等流程
- **减少重复造轮子**：把工程精力集中到语言设计本身
- **为未来自举铺路**：先建立稳定 IR 生成能力，再讨论完整生态

需要明确的是：

> 接入 LLVM **并不自动等于** 拥有“最强优化”。  
> 语言前端的类型信息、别名规则、内存模型、IR 生成质量、运行时设计，都会直接影响最终优化效果。

因此，bit 的工作不只是“把源码翻译成 LLVM IR”，而是要尽可能生成**高质量、优化友好、语义清晰**的 IR。

## 当前关注的问题

bit 目前重点关注这些基础问题：

- 语言的核心语法应该如何保持简洁与严肃
- 类型系统应该如何平衡表达力与可实现性
- 错误处理模型应该如何设计
- 内存与所有权语义要做到什么程度
- FFI 与 C 生态兼容策略如何定义
- 编译器前端与 IR 生成层如何分层
- 自举路线应当如何规划

这些问题的答案将逐步沉淀为：

- 语言说明文档
- 设计提案
- 编译器原型
- 测试样例
- 阶段性路线图
