# bit

> A new systems programming language built around LLVM IR.

**bit** 是一个面向系统软件的编程语言项目。  
它计划以 **LLVM IR** 作为核心后端依赖，在尽可能继承 LLVM 优化与代码生成能力的同时，重新设计一门更适合长期演进的基础语言。

bit 关心的不是“再做一门语法新奇的语言”，而是认真回答一个更困难的问题：

> 在现代编译基础设施已经足够强大的前提下，  
> 能不能重新设计一门兼顾 **性能、可控性、严肃性、可维护性、可移植性** 的基础编程语言？

## 项目状态

**当前阶段：早期原型 / stage1D**

bit 目前仍处于项目最早期。仓库当前重点不是提供一个立即可用的生产级编译器，而是：

- 收敛语言的目标与边界
- 明确语法、类型系统与语义方向
- 建立最小可工作的编译器原型
- 打通从源码到 LLVM IR 的基础链路
- 为未来自举与工具链扩展打基础

这意味着当前阶段的很多内容都应视为**草案**：

- 语法可能变化
- 类型系统尚未冻结
- 标准库尚未形成
- 编译器接口尚未稳定
- 仓库结构仍可能调整

## 什么是 bit

bit 是一门计划中的**系统编程语言**。

它的基本定位是：

- **靠近底层**
- **强调性能**
- **语义清晰**
- **工程可维护**
- **以 LLVM IR 为核心后端接口**
- **从一开始就为长期演进和自举做准备**

bit 不是 LLVM 的替代品。  
bit 计划建立在 LLVM 之上，把主要精力放在这些更值得重新设计的问题上：

- 语言本身的表达能力
- 类型系统与语义约束
- 内存与资源管理模型
- 模块系统与边界组织
- 错误处理策略
- FFI 与生态兼容方式
- 编译器前端与工具链架构

## 项目目标

bit 的长期目标包括：

### 1. 成为一门严肃的系统语言
它应该适合：

- 系统工具
- 基础设施软件
- 编译器与工具链
- 运行时与中间件
- 高性能服务
- 需要明确资源控制的软件

### 2. 尽可能继承 LLVM 后端能力
bit 计划通过 LLVM IR 获得统一的后端接口，使语言实现能把更多精力集中在前端设计与语义建模上，而不是重复建设完整后端。

### 3. 在“性能”和“语言现代性”之间取得更好的平衡
bit 不希望回到“只有裸能力、没有语言结构”的旧路径，也不希望为了高级抽象放弃对底层行为的控制。

### 4. 为自举与自托管预留路线
bit 从项目初期就会考虑：

- 最小可实现子集
- 稳定子语言边界
- 编译器自举路线
- 标准库与运行时的分层方式

### 5. 形成清晰、严格、可演进的规范
bit 追求的不是“无限灵活”，而是：

- 清晰
- 严格
- 可推导
- 可实现
- 可长期维护

## 设计原则

### 明确优于隐式

bit 倾向于减少那些“看起来方便、实际上让行为变得不透明”的机制。  
语义、类型行为、资源管理、错误传播都应尽量明确。

### 性能优先，但不牺牲结构

bit 追求高质量机器码，也追求对性能行为的强控制力。  
抽象不应天然意味着额外成本，零成本抽象应是重要方向之一。

### 语言前端负责语义，LLVM 负责后端

bit 不打算重复发明成熟后端。  
项目更希望把真正宝贵的复杂度投入到：

- 语法与语义设计
- 类型系统
- 约束表达
- IR 生成质量
- 工具链一致性

### 从最小闭环开始，而不是一开始追求大而全

bit 的路线会优先确保最小链路成立：

```text
source
  -> lexer
  -> parser
  -> AST
  -> semantic analysis
  -> LLVM IR
  -> object / executable
```

## 当前阶段

当前原型已经打通五个很小的阶段，并推进到 stage1D：

- stage0A：生成最小可验证的 LLVM IR 模块
- stage0B：把源码扫描成独立 token 流
- stage0C：把最小 token 流收束成独立 AST
- stage0D：把最小 AST lower 成真实 LLVM IR
- stage0E：在 AST 和 LLVM IR 之间插入独立语义检查
- stage1A：加入局部绑定、标识符表达式和最小名字解析
- stage1B：加入最小二元算术表达式和优先级解析
- stage1C：加入一元负号和最小负整数语义
- stage1D：加入函数参数、多函数声明与函数调用

目前 lexer 支持的最小 token 集包括：

- `fn`
- `return`
- `let`
- `i32`
- `(`
- `)`
- `{`
- `}`
- `->`
- `,`
- `:`
- `=`
- `+`
- `-`
- `*`
- `/`
- `;`
- 标识符
- 十进制整数字面量
- `EOF`
- `INVALID`

当前 parser 只支持这一段最小语法子集：

```bit
fn add(lhs: i32, rhs: i32) -> i32 {
    return lhs + rhs;
}

fn main() -> i32 {
    return add(1, 2);
}
```

成功解析后会生成最小 AST；失败时会在第一处错误直接停止，并报告 token 与行列号。

当前 sema 会在 IR 生成前做最小语义检查：

- 当前模块必须定义 `main`
- `main` 当前必须返回 `i32`
- `main` 当前不能带参数
- 函数名不能重名
- 函数调用会先走函数符号表，再单独检查局部变量符号表
- 局部绑定不能重名
- 标识符表达式必须先定义再使用
- 函数调用的实参数量必须和形参数量一致
- 函数调用的实参类型当前都必须匹配 `i32`
- 一元负号操作数当前必须是 `i32`
- 二元算术两侧当前都必须是 `i32`
- `return` 必须带表达式
- 十进制整数字面量必须落在 `i32` 正范围内
- 允许通过一元负号表达 `-2147483648`

当前 irgen 会把这棵最小 AST lower 成真实 LLVM IR，并只保留 backend 侧的防御性校验：

- 只接受 `i32`
- 支持模块内多个函数
- 先声明所有函数，再 lower 各自函数体
- 支持 `i32` 形参与 `call` lowering
- 支持一元负号 lowering
- 支持 `+`、`-`、`*`、`/` 的 LLVM 整数 lowering
- 遇到未知局部绑定或不完整 AST 会直接停止
- 生成结果仍会走 LLVM verify

## 构建计划

当前仓库使用 CMake 构建。下面的命令都可以从仓库根目录直接运行：

- `cmake -S . -B build`
  生成构建目录。
- `cmake --build build`
  编译当前原型工具。
- `./build/bitc examples/hello.bit -o hello.ll`
  运行当前最小前端链路：`lexer -> parser -> AST -> semantic analysis -> LLVM IR`。
- `./build/test_lexer examples/hello.bit`
  单独运行 stage0B lexer，打印 token 流与行列号。
- `./build/test_parser examples/hello.bit`
  单独运行 stage0C parser，打印稳定 AST dump。
- `./build/test_sema examples/hello.bit`
  单独运行 stage0E sema，验证最小语义规则。
- `./build/test_irgen examples/hello.bit hello.ll`
  单独运行 stage0D/stage0E 后的 lowering 链路，把 AST lower 成 `.ll` 文件。
- `ctest --test-dir build --output-on-failure`
  运行接入 `CTest` 的 lexer/parser/sema/irgen golden 回归测试。
